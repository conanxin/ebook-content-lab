# 《从大都到上都》子项目内容分析 v0.6

## 1. 当前页面定位

当前《从大都到上都》子项目已经不只是路线文件展示，而是一个证据型电子书内容项目，核心定位可以分成四层：

- 电子书路线地图：把书中从大都方向到上都方向的行走线索整理成 15 个路线段，并用地图、点位、GeoJSON 和 GPX 展示。
- 书中路线深读：新增 `book_overview.json`、`segment_reading_cards.json`、`place_index.json`、`book_themes.json` 后，页面开始解释每段路线在书中如何出现、有哪些地名、哪些内容需要注意。
- 复走参考：页面提供 GPX、连续徒步块、断点、补走、乘车和 `do_not_connect_in_gpx` 说明，但明确不作为未经核验的户外导航路线。
- 证据型阅读项目：每段路线仍以 `book_refs` 为事实依据，`chapter_refs` 只作为章节出处；新增细读内容也应围绕 OCR 文本、短摘和人工复核状态展开。

## 2. 当前内容数据层

- `route_segments.json`：路线事实底座。包含 15 段路线的起终点、途经地、路线概括、道路/水源/风险、证据引用、证据状态、复走状态和 GPX 连接规则。它是页面最重要的可审计数据，不应被随意重写。
- `book_overview.json`：项目总览层。解释本书路线结构、页面使用方式、徒步/混合/断点/GPX 的阅读方式，以及页面限制。
- `segment_reading_cards.json`：分段细读层。每段一张卡，提供书中场景概括、路线叙事、沿途地名、行走体验、读者应注意的细节、今日复走提示和证据出处。
- `place_index.json`：地名索引层。集中呈现重要地名、关联 segment、页码、书中上下文、坐标置信度和复核状态，适合解决 OCR 地名和历史地名核对问题。
- `book_themes.json`：主题阅读层。把路线结构、关口与边界、水源与湖淖、村镇与补给、徒步/乘车/断点、历史地名与现代定位等线索从分段数据中提炼出来。
- `route_walkable_blocks.json`：连续复走块。说明哪些 segment 可作为当前证据下的连续步行块，避免把 mixed 或断点段强行连成完整轨迹。
- `field_guide.md`：复走说明书。面向读者解释如何保守使用数据、哪些段不适合直接复走、GPX 如何使用、哪些问题需要人工核对。

## 3. 内容优势

- 15 段路线已经完整覆盖当前抽取的书中路线顺序。
- 每段路线都有 `book_refs`，并且经过证据审计与 evidence-resolved 清理。
- 页面现在具备 5 个 tab：总览、地图路线、路线细读、书中地名、复走说明。
- 地名索引覆盖了旧县村、八达岭关城、居庸关云台、不堡子村、塘子庙、馒头山村、石柱村、水泉淖尔、圆图淖尔、河北内蒙古分界线、滦河东岸沙丘等重点复核对象。
- 主题阅读已经开始把单段路线提升为跨段理解结构。
- GPX 导出有断点保护，`do_not_connect_in_gpx=true` 的段落不会被强连成连续轨迹。
- 页面口径总体保守，没有把项目表述为完整导航路线、官方路线或无需复核的精确轨迹。

## 4. 当前内容不足

部分段落的“路线细读”仍显得薄，主要原因是当前 `segment_reading_cards.json` 大多从已有 `route_segments.json` 字段和 `book_refs` 派生，尚未充分利用 OCR 文本中的叙事细节。较明显的段落包括：

- `seg-001`：有出发方向、皂甲屯、餐馆等证据，但还可以解释为什么从健德门出发构成整条路线的叙事开端。
- `seg-003`：居庸关、八达岭、水关长城、延庆断点是关键内容，但 mixed、gap_after 和现代路况复核需要更清楚的读者说明。
- `seg-008` 和 `seg-009`：老掌沟前后涉及车辆接应与补走逻辑，是全项目最容易被误读成连续徒步的部分，需要更细的“书中怎么走、哪里不是徒步”的拆解。
- `seg-010`、`seg-011`、`seg-014`、`seg-015`：这些段落有乘车、补走或桥北开始等问题，细读层还可以更明确区分书中步行、乘车、补走和地图展示边界。

地名索引仍有不足：

- 一些地名目前只是索引项，缺少更充分的历史语境说明，例如不堡子村、馒头山村、圆图淖尔、滦河东岸沙丘。
- 部分 OCR 标题页地名存在误识别或疑似误识别，索引中已经标记 `needs_review`，但还没有人工复核结论。
- 坐标置信度与地名复核状态需要在页面上更明确地连起来，避免读者把 approximate 坐标理解为已核定地点。

主题阅读仍可展开：

- “关口与边界”可以进一步说明居庸关、八达岭、河北与内蒙分界点在路线阅读中的结构作用。
- “水源与湖淖”可以增加湖淖地名 OCR 复核说明，尤其是水泉淖尔、圆图淖尔。
- “徒步、乘车与断点”应成为核心主题，单独解释 mixed 与 `do_not_connect_in_gpx` 的区别。
- “历史地名与现代定位”需要继续强调现代坐标只是辅助，不是路线证据。

复走说明仍可改进：

- 当前说明偏文字化，后续可以增加“可走 / 不可走 / 需核验”表格。
- 对 `seg-009` 这类 mixed 但进入 GPX track 的特殊情况，需要在复走说明中单列解释。
- 对只导出 waypoint 的段落，可以列出“不应强连原因”和“人工核验问题”。

OCR / 坐标 / 断点仍需人工复核：

- OCR 地名重点：旧县村、八达岭/居庸关、不堡子村、塘子庙、馒头山村/石柱村、水泉淖尔/圆图淖尔、河北内蒙古分界线、滦河东岸沙丘。
- 断点重点：`seg-003`、`seg-008`、`seg-009`、`seg-010`、`seg-011`、`seg-013`、`seg-014`、`seg-015`。
- 坐标重点：`modern_followability=needs_field_check` 的段落，以及 `coordinate_confidence=approximate` 的历史地名。

## 5. 页面交互问题

本次发现的问题是 `#reading-detail` 导致线上进入 NotFound 页面。根因是应用本身使用 hash route：

- 首页：`#/`
- 子项目：`#/projects/dadou-shangdu`

如果页面内部再使用裸 hash anchor，例如 `#reading-detail`，浏览器会把整个 hash route 覆盖掉，实际 URL 变成：

- `https://conanxin.github.io/ebook-content-lab/#reading-detail`

应用路由层会把 `reading-detail` 当作顶层路由解析，因此显示“未找到页面：reading-detail”。

修复方向是使用 hash route 内部的 query 参数控制 tab：

- `#/projects/dadou-shangdu?tab=overview`
- `#/projects/dadou-shangdu?tab=map`
- `#/projects/dadou-shangdu?tab=reading-detail`
- `#/projects/dadou-shangdu?tab=place-index`
- `#/projects/dadou-shangdu?tab=field-guide`

这种方式保留了 hash route 的顶层路径，不会覆盖项目路由。旧链接如 `#reading-detail` 应自动跳转到规范 URL，避免已传播链接直接 404。

## 6. 下一步内容优化建议

### P0

- 修复 hash tab 404。
- 保证 5 个 tab 可直接访问。
- 保证旧 hash 自动跳转。

### P1

- 每段增加“书中这一段为什么重要”。
- 地名索引增加更具体的“书中上下文”和“复核状态”说明。
- 主题阅读增加段落关联和证据，尤其是关口、边界、湖淖、断点主题。
- 复走说明增加“可走 / 不可走 / 需核验”表格。

### P2

- 增加阅读模式，弱化地图，强化按书中页码和主题浏览。
- 增加 segment 比较视图，例如 walked 与 mixed、GPX track 与 waypoint-only 的对照。
- 增加 GPX 使用教学，说明哪些 App 会自动连线以及为什么需要避免误读。
- 增加人工复核提交模板，把 `manual_review_tasks_v0.6.csv` 转化为更易填写的网页或 issue 模板。

## 7. 不应做的事

- 不公开 OCR 全文。
- 不复制长段原文。
- 不把 approximate 坐标写成确定地点。
- 不把页面称为完整导航路线。
- 不用现代地图反推或改写书中路线。
- 不把 `chapter_refs` 当作路线事实证据。
